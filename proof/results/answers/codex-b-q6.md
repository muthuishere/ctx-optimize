In BFQ, the budget timeout is the per-in-service-queue deadline for consuming its budget: `bfqq->budget_timeout = jiffies + bfqd->bfq_timeout * timeout_coeff`, set by `bfq_set_budget_timeout()` when a queue becomes in service. The default base timeout is `bfq_timeout = HZ / 8`, and `timeout_sync` stores user input as `bfqd->bfq_timeout` in jiffies. See `block/bfq-iosched.c:203`, `block/bfq-iosched.c:3285`, `block/bfq-iosched.c:3297`, `block/bfq-iosched.c:7507`, `block/bfq-iosched.c:7523`.

The direct checker is `bfq_bfqq_budget_timeout()`, which returns `time_is_before_eq_jiffies(bfqq->budget_timeout)`. It is not a dedicated timer; it is checked on arrivals, completions, and idle-timer expiry. `bfq_may_expire_for_budg_timeout()` adds the policy gate. See `block/bfq-iosched.c:4456`, `block/bfq-iosched.c:4460`, `block/bfq-iosched.c:4473`.

When it fires, the in-service queue is expired with reason `BFQQE_BUDGET_TIMEOUT` (`budget took too long to be used`). Callers invoke `bfq_bfqq_expire(..., BFQQE_BUDGET_TIMEOUT)`, which recalculates the budget, calls `__bfq_bfqq_expire()`, then the queue is either removed from busy state if empty or requeued if it still has backlog, and BFQ resets the in-service state via `__bfq_bfqd_reset_in_service()`. For this timeout reason, `entity->service` is reset to `0`. See `block/bfq-iosched.h:908`, `block/bfq-iosched.c:6465`, `block/bfq-iosched.c:4415`, `block/bfq-iosched.c:3890`, `block/bfq-iosched.c:3915`, `block/bfq-iosched.c:3928`, `block/bfq-iosched.c:3946`, `block/bfq-iosched.c:4421`.
